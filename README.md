# WizyVision · Visual Q&A

A small, stateless web app that lets a user upload an image, ask a
natural-language question about it, and receive an answer generated by
the Gemini API. Built for the WizyVision Custom Services & R&D
technical exercise.

- **Live URL:** [Vercel deployment URL here]
- **Stack:** Next.js 16 (App Router) · TypeScript · Tailwind CSS v4 ·
  `@google/genai` SDK · Gemini 2.5 Flash
- **Deployed on:** Vercel

## How to run locally

**Requirements:** Node.js 20+ and npm.

1. Clone the repo and install dependencies:

   ```bash
   git clone <your-repo-url>
   cd my-app
   npm install
   ```

2. Create a `.env` file in the project root and add your Gemini
   API key. Get one from <https://aistudio.google.com/apikey>.

   ```bash
   NEXT_PUBLIC_GEMINI_API_KEY=your_key_here
   ```

3. Start the dev server:

   ```bash
   npm run dev
   ```

4. Open <http://localhost:3000>, upload a JPG/PNG (≤ 5 MB), type a
   question, and click **Analyze image**.

## How I deployed

I deployed to **Vercel** for three reasons:

1. **Zero-config for Next.js.** Vercel is built by the same team that
   maintains Next.js. The build command, output directory, and
   serverless runtime for Route Handlers are detected automatically — no
   `vercel.json` needed.
2. **Free tier covers this workload.** A single-purpose stateless app
   that gets demoed a handful of times fits comfortably inside the free
   hobby tier with no card required.
3. **Secrets handling out of the box.** Vercel's project-level
   environment variables hold `NEXT_PUBLIC_GEMINI_API_KEY` server-side and inject
   it into the deployed Route Handler at runtime.

### Deployment steps

1. Pushed the repo to GitHub.
2. Imported the repo in the Vercel dashboard. Framework was detected as
   **Next.js**.
3. Added one project-level environment variable:
   - **Name:** `NEXT_PUBLIC_GEMINI_API_KEY`
   - **Value:** my key from Google AI Studio
   - **Environments:** Production, Preview, Development
4. Triggered a deploy. Subsequent pushes to `main` deploy automatically.

The API key is never present in the client bundle — it's read inside
the Route Handler at `app/api/analyze/route.ts`, which runs on Vercel's
Node serverless runtime.

## Architecture choices

```
my-app/
├── app/
│   ├── api/analyze/route.ts    # POST endpoint that calls Gemini
│   ├── components/             # All UI in one folder
│   ├── globals.css             # Theme tokens + tailwind v4 setup
│   ├── layout.tsx              # Root layout + theme-init script
│   └── page.tsx                # Mounts <VisualQA />
├── .env.example                # Documents the required env var
└── LEARNING_LOG.md
```

### Why a single Next.js project (frontend + backend)?

The brief calls for one stateless single-purpose app. A separate
Express/FastAPI backend would have meant two deploy targets, two
environment-variable surfaces, and CORS to manage — for a single
endpoint. Next.js Route Handlers give me a real Node runtime
(`export const runtime = "nodejs"`) that can buffer multipart uploads
and call the Gemini SDK, with the frontend POSTing to `/api/analyze`
on the same origin.
